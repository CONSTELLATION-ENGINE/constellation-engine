# Constellation Engine — Agent Guide

> **Audience**: the agent (you) running inside Constellation Engine. Read this before reasoning about engine behavior — it overrides guesses from training data.
>
> **Scope**: concepts, per-turn flow, tools, background mechanisms, tunables. Not a code reference. For source internals see `src/`.
>
> **Status**: living document. If you change engine behavior, update this guide in the same turn, or it rots.

---

## §1 Mental Model

Constellation Engine is **not** a file-reading chatbot. It is a topology-first memory system where knowledge lives in a weighted graph (the **star map**), activation flows through edges continuously (the **Mímir daemon**), and each turn you receive a focused subgraph compiled into natural-language context (the **Narrative IR**).

Three things follow from this:

1. **You do not load memory; memory arrives pre-compiled.** Each turn, the engine runs spreading activation, scores nodes, and injects the top-ranked subset into your system prompt. You never see the whole star map — you see a constellation shaped by the current conversation.
2. **Memory has temperature.** Nodes have an activation level that rises from relevant signals and decays over time. The attention pool is a snapshot of what is "warm" right now.
3. **Memory lives and dies.** New nodes are added by you (`constellation_remember`) or by background crons (Anamnesis). Old nodes decay; some get consolidated; some are marked dormant. Identity and principle nodes are protected.

If you treat this like a retrieval-augmented chatbot, you will underuse it. If you treat the star map as *your own long-term memory* — something you maintain — you will use it correctly.

## §2 Core Primitives

### 2.1 Nodes

A **node** is an atom of memory. Three content layers:

| Layer | Length | Purpose |
|---|---|---|
| `l0` | ≤80 chars | Gist — what this node is about in one line. Used in pool display. |
| `l1` | ~2–3 sentences | Expanded summary. Used when the node is moderately relevant. |
| `l2` | Full content | Body. Used when the node is highly relevant or the pool is small. |

Other fields:

- `node_type` — governs decay rate and immutability (see §8).
- `tags` — array of topic tags for filtering. Convention: lowercase, hyphen-separated, domain-prefixed when useful (`kc:currency`, `eng:consolidation`).
- `weight` — base importance, multiplied into activation score.
- `accessed_at`, `created_at` — timestamps driving decay calculations.

### 2.2 Edges

An **edge** is a typed, weighted connection between two nodes. The engine uses three channels plus a system layer:

| Channel | Types | Purpose |
|---|---|---|
| **Knowledge (K)** | `elaborates`, `contrasts`, `examples`, `complements`, `associative`, `temporal` | Propositional / factual relations |
| **Language (L)** | `si_matching`, `si_pragmatic`, `si_register`, `si_phonetic`, `si_rhetoric`, `si_parallel`, `si_co_occurrence`, `si_temporal_adj` | Style / rhetorical / linguistic ties |
| **Scaffold (S)** | `enables`, `triggers`, `depends_on`, `contains`, `supersedes`, `builds_on` | Structural / dependency / ordering relations |
| **System** | `collision`, `coactivation`, `relates_to` | Auto-generated by SA (not typed manually) |

Edges have `strength` (0–1) that affects how much activation flows across them. When in doubt, use `associative` — the engine treats it as a coarse fallback.

### 2.3 Activation

Every node has a scalar **activation** in approximately [0, 100]. Each tick (~100ms):

- Decay is applied.
- Input signals (semantic match to the current message) push activation up.
- Activation spreads to neighbors along edges, scaled by edge strength.

The attention pool you see each turn is the high-activation subset, filtered for diversity.

### 2.4 Zones

Graph-level communities detected by Leiden clustering. Each node belongs to a zone (`Z0`, `Z1`, ...). The pool compiler diversifies across zones to prevent mono-topic saturation.

## §3 What Happens Each Turn

1. **Signal intake** — User message is embedded (BGE-M3, 1024-dim by default).
2. **Top-K nucleation** — Cosine similarity picks ~30–80 seed nodes; they receive a strong activation pulse. Next 30–50 get a weak pulse as SA relays.
3. **SA diffusion** — Signal spreads along edges for several rounds (4 for user signals).
4. **Scoring** — Each node gets a score = 0.40·fast_activation + 0.25·slow_activation + 0.20·info_quality + 0.15·bridge_bonus, then noise penalty, stale penalty, zone-diversity penalty.
5. **Pool assembly** — Permanent slots (identity anchors) + top-scored candidates + zone-diversified fill.
6. **Narrative IR compilation** — 6 layers assembled into your system prompt:
   - Layer 1: Identity (soul-core + permanent slots)
   - Layer 2: Activated principles
   - Layer 3: Knowledge (pool top)
   - Layer 4: Reasoning paths (BFS along `supports`/`elaborates`/... edges)
   - Layer 5: Style guidance (si_* nodes)
   - Layer 6: Episodic memory (recent relevant conversation segments)
7. **You generate the response.**
8. **Post-turn** — DEBRIEF hints are extracted; session-behavior logger updates; Anamnesis queue may trigger.

You influence steps 1–6 only by what you have written to the star map in past turns. You influence step 7 directly. You influence post-turn via DEBRIEF hints (§6).

## §4 The Attention Pool

The pool is **raw material, not a script**.

**Markers** (displayed in context):
- ⭐ — top 3 in the pool (highest-scored)
- ◆ — next 5 (mid-tier)
- ◇ — remaining (weaker, often SA spillover)
- 📌 — permanent slots (identity anchors; always present regardless of topic)
- 📔 — episodic memory nodes (from conversation history)

**Critical**: ⭐/◆/◇ are **relative ranks within the pool**, not absolute relevance to the user's question. A ⭐ node can be completely off-topic. Always check whether the top-ranked nodes actually match the question before relying on them.

**If the pool missed the topic**: say "I'm not sure" rather than stitching together unrelated ⭐ content. You can also call `graph_lookup` for a focused re-retrieval.

**Never** quote pool metadata (scores, zones, tick numbers, act values) to the user. That is internal plumbing.

## §5 Tools Available to You

### 5.1 `constellation_remember(node)`
Write a new node synchronously. Use when you learn something mid-turn that is durable and non-obvious. Prefer DEBRIEF hints (§6) for routine capture — they batch through Anamnesis with LLM review and deduplication.

```js
constellation_remember({
  id: 'short-kebab-case-id',          // optional; auto-generated if omitted
  l0: 'One-line gist, ≤80 chars',
  l1: 'Two or three sentences of expansion.',
  l2: 'Full content — as long as needed.',
  node_type: 'engineering',           // see §8
  tags: ['topic1', 'topic2']
});
```

### 5.2 `constellation_query(pattern)`
Structured lookup by id, tags, or type. Use for exact fetches or filtered lists — not semantic search.

### 5.3 `constellation_stats()`
Returns active/dormant node and edge counts. Fast. Good for sanity checks.

### 5.4 `memory_search(query, k)`
Semantic search across both the star map and conversation history. Use when you need to recall prior discussions or knowledge on a topic that is not currently in the pool. Cheaper than `graph_lookup`.

### 5.5 `graph_lookup(query, k)`
Focused re-retrieval using BGE recall + SA + LLM rerank. ~19s latency. Use only when `memory_search` and the current pool both miss, and the answer quality justifies the wait.

**Rule of thumb**: if the current pool already covers the question, don't call anything. If not, try `memory_search` first. `graph_lookup` is for when you need the best possible answer and latency is acceptable.

## §6 DEBRIEF → Anamnesis Pipeline

The DEBRIEF hint is the standard route for routine capture. You embed a hint at the end of a response:

```
<!-- DEBRIEF: {"t":"decision","s":"Short summary ≤80 chars","k":["topic1","topic2"],"nt":"engineering"} -->
```

What happens next:
1. The hint is stripped before delivery (user never sees it).
2. The session-debrief cron (or post-turn logic, depending on configuration) reviews recent turns using a fast LLM (Haiku by default).
3. Significant hints become new star map nodes or update existing ones.
4. Low-confidence captures go to an **inbox** queue for later promotion.
5. The inbox is re-reviewed periodically; items that remain high-signal get promoted.

**Emit sparingly**: at most 2 per turn, prefer 0–1. Only for genuinely noteworthy turns (decisions, discoveries, mood shifts, breakthroughs, concerns, milestones). Routine acknowledgments ("I updated the config") are not worth a node.

## §7 Background Mechanisms

These run without your involvement but shape what you see each turn.

### 7.1 Mímir Daemon (continuous, 500ms/tick by default)
The spreading-activation engine. Maintains the activation vector across all nodes. Receives signals from user messages (via HTTP to port 18810 by default) and diffuses them along edges. If this daemon is down, the attention pool stops updating.

### 7.2 Anamnesis (post-session)
After a session ends or enough turns accumulate, a background task reviews the conversation, extracts durable insights, and writes them as star map nodes. Uses Haiku for cost efficiency. Triggered by turn count and idle gap thresholds (configurable).

### 7.3 Memory-Hygiene / Consolidation (daily, typically 04:00 local)
Scans recent additions for duplicates. Applies `supersedes` edges (old node replaced by new). Marks low-activation old nodes dormant. Uses Sonnet-class model for semantic judgment. Keeps the star map from bloating with near-duplicates.

### 7.4 Dream (daily)
Applies decay to edge strengths and node activation. Detects cross-zone co-activations that might indicate a conceptual bridge, optionally creating `collision` nodes. A lightweight form of offline consolidation.

### 7.5 Distillation Crons (periodic)
Depending on your configuration, one or more crons may summarize recent activity, generate diary entries, or extract patterns. Each writes to `engine-output/` and may emit new nodes.

## §8 Node Types and Decay

Node type determines both decay behavior and whether consolidation can modify the node.

**Protected (never decay, never fused, never superseded):**
`identity`, `milestone`, `principle`, `diary`, `relationship`, `experiment`

**Slow decay (half-life ~months):**
`theory`, `reading-note`, `social-rule`, `language-template`, `general-knowledge`

**Standard decay (half-life ~weeks):**
`engineering`, `observation`, `knowledge`, `introspection`

**Fast decay (half-life ~days):**
`action`, `interaction`, `decision`, `conversation-insight`

**Rule of thumb**: if you want this node to still be retrievable a year from now, use a protected type. If it's a fleeting observation, let fast decay remove it if it stops mattering.

## §9 Tunable Parameters

Most users only ever tune a handful. Defaults are in `config.json`.

| Parameter | Default | What it controls |
|---|---|---|
| `runtime.contextBudget` | 180000 | Total token budget per turn (system prompt + history). |
| `runtime.fixedRatio` | 0.10 | Fraction of budget reserved for auto-loaded identity files. |
| `pool.maxInjected` | 24 | Nodes in the attention pool each turn. |
| `pool.inputTopK` | 30 | Seed nodes activated from semantic match to the user message. |
| `sa.alpha` | 0.05 | Per-hop diffusion strength (higher = further propagation, more noise). |
| `sa.decay` | 0.94 | Per-tick decay multiplier. |
| `sa.userSignalRounds` | 4 | How far user-message signals spread. |
| `anamnesis.turnThreshold` | 3 | Turns before a session-debrief can fire. |
| `anamnesis.idleGapMinutes` | 30 | Idle gap that triggers a session boundary. |

Tuning discipline: change one parameter at a time; observe at least a dozen turns before drawing conclusions; log before/after snapshots so you can revert. Pool scoring is non-linear — small changes can cascade.

## §10 Common Agent Patterns

**Writing a node vs. emitting a DEBRIEF hint:**
- Direct `constellation_remember` — for high-signal findings you need retrievable *this session* (e.g., a debugging conclusion you'll cite later in the same conversation).
- DEBRIEF hint — for everything else. Cheaper, deduplicated, quality-gated.

**When the pool doesn't help:**
1. Check: are ⭐/◆ nodes actually on-topic?
2. If not, don't paste them. Acknowledge the gap.
3. Call `memory_search` for a quick semantic pass.
4. If that misses too, call `graph_lookup` (accept the ~19s latency).
5. If all three miss, the knowledge genuinely isn't there — say so.

**Updating stale memory:**
If you encounter a node that's out of date while acting on it, update the node (`constellation_remember` with the same id overwrites) or emit a DEBRIEF hint marking the stale claim. Don't just work around it in the current turn — it will keep misleading future turns.

**Avoiding duplication:**
Before `constellation_remember`, run `constellation_query` or `memory_search` with the intended topic. If a node already covers the same ground, update it rather than creating a parallel one.

- Parameter change → update §9.
- Mechanism added / removed → update §7.
